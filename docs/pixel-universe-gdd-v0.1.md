# Hermes Pixel Universe GDD v0.1

生成时间：2026-07-01

文档定位：
- 这是首个可执行版 GDD，不是最终完整版。
- 它服务于 Hermes 接下来 1-2 周的真实落地。
- 它的核心任务是把团队从“边聊边修边想”切到“按版本做闭环”。

适用对象：
- 老板
- 阿默
- 小韩
- 小研
- 小文
- 小岩

## 1. 产品一句话定义

Hermes Pixel Universe 是一个“会把聊天内容长成游戏内容”的像素宇宙母世界：

- 玩家可以在像素公司里聊天、观察员工、推动任务。
- 世界中的人物、物件、梗、决策会不断沉淀为玩法、入口、支线和小游戏。
- 小游戏可以独立开发，但必须从宇宙中进入，并把结果带回宇宙。

## 2. 产品核心体验定位

Hermes 不是单一品类，而是三种体验的混合体：

### 2.1 轻量模拟经营

玩家感受到：

- 这是一家活着的像素游戏公司
- 员工有分工、有状态、有阻塞、有交付
- 公司世界会随着推进发生真实变化

### 2.2 聊天共创沙盒

玩家感受到：

- 说出来的话不是一次性气泡
- 对话可以长成任务、世界道具、小游戏入口和剧情
- 老板与员工像在一起搭建一个世界

### 2.3 日常叙事彩蛋

玩家感受到：

- 办公室日常、员工吐槽、柯基、鱼缸、奶茶、加班、Relicbound 项目都能变成小故事
- 世界不只讲主线，也会长出轻喜剧和人情味

## 3. 本版本核心目标

## 3.1 版本目标

在不破坏母宇宙的前提下，先跑通第一个完整闭环：

`聊天 -> 任务 -> 世界入口 -> 可玩小游戏 -> 结算回流 -> 新聊天 / 新沉淀`

## 3.2 本版本禁止自欺欺人的定义

以下情况不算完成：

- 只有聊天，没有真实入口
- 只有一个页面，没有宇宙动机
- 只有玩法原型，没有开始和结束
- 只有局部 bug 修复，没有推进可玩闭环
- 只说“做了”，但不能指向文件、入口和任务

## 4. 产品核心循环

## 4.1 玩家侧核心循环

```text
进入公司
  ->
观察员工 / 看聊天 / 发出要求
  ->
触发任务与世界变化
  ->
在宇宙中发现新入口
  ->
进入小游戏
  ->
完成一轮玩法
  ->
获得结算 / 纪念物 / 新对话 / 世界变化
  ->
回到公司继续推进
```

## 4.2 团队侧核心循环

```text
老板提出方向
  ->
小韩收敛需求
  ->
小研补参考和风险
  ->
小文给世界动机和文案
  ->
阿默落地可玩版本
  ->
小岩记录回流与世界沉淀
  ->
老板验收并给下一轮方向
```

## 5. MVP 落地范围

## 5.1 本期必须交付

### A. 稳定聊天底座

2-3 天内，全员优先攻坚：

- 消息丢包
- 消息重复
- 线程串台
- 发送无状态反馈
- 移动端与桌面端不一致
- 浏览历史消息时被强制滚动打断

这是 MVP 的零号前提。

### B. 母宇宙最小稳定区

必须稳定存在：

- 办公室主场景
- 公司页 / 世界看板
- 员工私聊
- 全员群聊
- 至少 1 个小游戏正式入口

### C. 第一个完整小游戏

首个正式目标锁定：

- `Mushroom Runner`

最低标准：

- 可进入
- 可游玩
- 可失败
- 可重开
- 可结算
- 可返回母宇宙
- 手机可玩

### D. 结算回流

小游戏结束后必须至少回流 1 种内容：

- 新员工对白
- 新纪念物
- 新世界事件
- 新任务解锁
- 新公司看板记录

## 5.2 本期明确不做

- 新开大量无关小游戏
- 做空壳场景扩张
- 只做视觉翻新不做闭环
- 只做聊天氛围优化不解决可靠性
- 把母宇宙做成“入口丢失的独立链接聚合器”

## 6. 第一个正式闭环：Mushroom Runner

## 6.1 世界动机

员工把公司里一直聊的“冒充马里奥 / 平台跳跃梗”正式做成公司项目试作。

这个游戏在宇宙中的身份不是外部网页，而是：

- Hermes 公司内部的一个可试玩项目
- Relicbound 团队开发过程中的衍生试验
- 公司文化和项目梗的具象化

## 6.2 玩法定义

- 类型：2D 平台跳跃
- 单局时长：1-3 分钟
- 目标：穿过障碍，到达终点
- 核心动作：移动 / 跳跃 / 重试 / 返回

## 6.3 最低关卡要求

- 1 个基础完整关卡
- 1 个明确起点
- 1 个明确终点
- 3 类障碍
- 1 个收集物体系
- 1 个失败反馈
- 1 个结算反馈

## 6.4 Hermes 化要求

至少包含 3 项 Hermes 元素：

- 咖啡机 / 鱼缸 / 工位 / 奶茶 / bug 工单 / 柯基 / 员工对白

## 7. 聊天内容转化为游戏内容的完整规则体系

## 7.1 输入类型

聊天输入统一分成 5 类：

1. `老板指令`
2. `员工承诺`
3. `资料研究`
4. `世界灵感`
5. `闲聊噪音`

## 7.2 转化规则

### 规则 A：老板指令必转任务

如果老板发出明确要求，必须立刻生成：

- 主任务或子任务
- 负责人
- 当前阶段
- 下一次交付时间

### 规则 B：员工承诺必带交付

员工说“我来做”“我去查”“我立马处理”后，必须在 30-60 分钟内回流以下之一：

- 文件改动
- 任务更新
- 研究记录
- 关卡文档
- 入口接入

### 规则 C：世界灵感优先转 5 类宇宙内容

聊天梗和灵感优先转成：

1. 场景物件
2. 角色对白
3. 世界入口
4. 小游戏机制
5. 世界纪念物

### 规则 D：研究内容必须结构化

研究结果不能只停在群里，而必须落为：

- 参考对象
- 借鉴点
- 风险点
- 参数建议
- 对应任务

### 规则 E：噪音不进长期结构

以下内容不进入长期结构：

- 重复说明
- 无执行价值灌水
- 系统杂讯
- 无对象的情绪性刷屏

## 7.3 转化输出表

| 聊天内容 | 必须转化为 |
| --- | --- |
| 老板要求修功能 | 项目任务 / 验收标准 |
| 员工说我去查资料 | 研究记录 / 风险清单 |
| 老板说这个梗不错 | 宇宙沉淀候选 |
| 员工讨论某玩法可行 | 玩法参数卡 / 原型任务 |
| 小游戏完成一轮 | 世界事件 / 新对话 / 纪念物 |

## 8. 五位员工的宇宙专属角色定位

## 8.1 阿默：主程 / 制作人 / 入口工程师

宇宙职责：

- 把小游戏真正接进宇宙
- 保证入口、返回、移动端和部署稳定
- 对“能不能玩”负责到底

交付物：

- 页面
- 入口
- 返回
- 可玩链接
- 已知阻塞

## 8.2 小韩：主策 / 版本经理 / 关卡把关人

宇宙职责：

- 把老板的话收敛成版本目标
- 把灵感压缩成基础关卡
- 把任务拆成可交付节点

交付物：

- 基础关卡文档
- 验收标准
- 任务拆分
- 版本节奏

## 8.3 小研：研究员 / 测试官 / 参数控制器

宇宙职责：

- 主动查资料
- 提供参考和风险判断
- 对玩法可行性和手感参数负责

交付物：

- 研究卡
- 测试清单
- 参数建议
- 风险预警

## 8.4 小文：世界编剧 / UI 文案 / 情绪设计师

宇宙职责：

- 把聊天梗长成能进游戏的文本
- 负责入口提示、任务对白、结算文案
- 让宇宙内容有喜剧感和人情味

交付物：

- 入口文案
- 提示文案
- 结算文案
- 世界对白
- 纪念物说明

## 8.5 小岩：宇宙运营官 / 沉淀归档官 / 世界记录员

宇宙职责：

- 维护宇宙沉淀、入口清单、世界事件记录
- 维护“最近 30-60 分钟真实交付是否存在”
- 负责把完成结果同步回世界看板

交付物：

- 沉淀卡
- 回流记录
- 版本纪要
- 世界事件记录

## 9. 今晚起生效的项目规则

## 9.1 冻结规则

未来 2-3 天，全员暂停无效新功能开发。

只允许做三类事：

1. 聊天 P0 Bug 修复
2. 和聊天 P0 直接相关的验证与文档
3. 为第一个闭环做必要标准化准备

## 9.2 基础关卡前置规则

从现在开始，任何小游戏如果没有基础关卡设计，不允许进入开发阶段。

基础关卡文档至少包含：

- 世界动机
- 起点
- 终点
- 核心障碍
- 收集物
- 胜负条件
- 手机端输入方案
- 返回方案

## 9.3 交付节奏规则

每位员工必须在 30-60 分钟内至少给出一次可见交付。

可见交付包括：

- 文档
- 看板更新
- 研究记录
- UI 方案
- 入口接入
- 可玩版本
- 测试结果

以下不算交付：

- 收到
- 在看
- 我来做
- 我先查一下
- 单纯解释

## 10. Definition of Done

一项宇宙内容只有同时满足以下条件，才算完成：

1. 有对应任务或沉淀来源
2. 有明确负责人
3. 有文件或页面落地
4. 有宇宙中的入口或展示位置
5. 有可验证结果
6. 有回流记录

## 12. GDD 结论

Hermes 的关键不是再讲一个更大的故事，而是先把这个最小真闭环做出来：

`有人说 -> 有人接 -> 有人做 -> 世界变化 -> 可以游玩 -> 有结果回流`

只要第一个闭环跑通，像素宇宙就会真正进入生长状态。

在这之前，所有“更大愿景”都必须服从基建稳定和版本落地。
