# Hermes Pixel Universe 产品需求书 v1

生成时间：2026-06-30

适用对象：
- 老板
- 小韩（策划主编）
- 阿默（主程 / 制作人）
- 小研（玩法研究 / 测试）
- 小文（叙事 / 文案 / UI 体验）

## 1. 背景问题

Hermes Pixel Office 已运行半个月，但团队仍未交付一个真正可玩的完整游戏。当前主要问题不是“完全没有想法”，而是：

- 员工持续被 bug 修复、聊天响应、临时需求打断。
- 项目目标停留在“做点什么看看”，没有被收敛成明确版本。
- 已有灵感很多，但没有进入“主世界入口 -> 可玩闭环 -> 可验收交付”流程。
- 每个人都在参与，但没人对“一个完整游戏必须包含什么”负责到底。
- 看板有任务，世界里有聊天，但没有形成真正的游戏生产线。

这份需求书的目的，是把 Hermes 从“像素公司聊天系统”升级为“会持续长出游戏内容的像素宇宙母世界”。

## 2. 产品定义

Hermes Pixel Universe 不是单个小游戏，也不是一个空壳世界。

它是一个由“像素公司主世界”向外扩展的游戏宇宙，包含三层：

1. 母世界：
   Hermes 像素办公室、公司大厅、老板办公室、街区、别墅外部、厕所、鱼池、水吧等。
2. 内容世界：
   由聊天、员工性格、公司物件、项目梗生长出来的剧情、道具、任务和事件。
3. 玩法世界：
   从母世界入口进入的可玩小游戏、关卡、战斗、平台跳跃、探索等。

补充架构原则：

- 每个小游戏可以单独开发、单独迭代、单独测试。
- 但每个小游戏的正式入口，必须存在于 Hermes 像素宇宙内部。
- 玩家必须通过宇宙中的交互物件、入口区域、角色对话或世界设施进入小游戏。
- 小游戏页面可以独立，母宇宙不能因此被小游戏开发过程频繁破坏。
- 独立开发的目的，是保护母世界稳定，而不是把小游戏做成与宇宙无关的外部孤岛。

最终目标不是每天多一个 demo，而是让玩家进入 Hermes 后，能感受到：

- 这是一个活着的公司世界。
- 员工、聊天、任务、物件、小游戏是同一个宇宙的一部分。
- 任何新内容都应该优先长在这个宇宙上，而不是脱离母世界单独长一个孤岛。
- 老板说出的明确需求，会在一段时间后真实长成世界里的内容，而不是只留在聊天记录里。

## 3. 产品目标

### 3.1 总目标

在接下来的版本推进中，Hermes 必须先交付 1 个真正完整、可进入、可退出、可在手机上玩的小游戏，再扩展到 3 个第一批小游戏，最后形成可持续扩展的宇宙内容生产线。

### 3.2 阶段目标

#### M1：先交付第一个完整游戏

选择一个最适合现有资产和世界观的方向，先做成完整版本。

“完整”至少包括：

- 有明确入口
- 有开始和结束
- 有可重复游玩的一轮玩法
- 有基础目标或胜负条件
- 有反馈、文案、结算
- 能从主世界进入
- 能安全返回主世界
- 手机端可玩
- 员工能解释它为什么存在于 Hermes 宇宙中

#### M2：扩展第一批 3 个小游戏

首批只允许做 3 个，不能无限开坑：

1. `Mushroom Runner`
   原“水管工冒充者”，统一命名。
2. `柯基追鱼`
   基于柯基、鱼缸、办公室障碍生成的追逐闪避玩法。
3. `奶茶外卖`
   基于员工、路线、公司空间生成的配送/识别玩法。

#### M3：建立宇宙扩展规则

后续所有小游戏和外部区域，都必须遵守以下规则：

- 从母世界长出来
- 复用现有角色、物品、对话或项目梗
- 有对应负责人
- 有明确验收
- 有返回主世界的路径
- 有真实沉淀，不允许“做完了但世界里什么都没变”
- 可以单独开发，但入口必须锚定在宇宙内

#### M4：建立连续交付节奏

Hermes 团队不能再以“有在聊”“有在修 bug”作为推进证明。

从这份 PRD 生效开始，主线开发必须进入固定节奏：

- 每 30 到 60 分钟至少有一次可见交付
- 交付必须对老板可理解、可检查
- 长时间只有讨论、不见成果，视为项目失速

## 4. 本期范围

本期只做以下内容，不再无边界扩散：

### 4.1 主世界范围

- 办公室
- 公司大厅 / 手机公司页
- 宇宙广场 / Companyverse 主入口
- 小游戏入口区

### 4.2 第一批小游戏范围

- `Mushroom Runner`
- `柯基追鱼`
- `奶茶外卖`

### 4.2.1 开发边界

为了不破坏原始母宇宙，第一批小游戏允许使用“独立页面 / 独立目录 / 独立资源”的方式开发。

但必须同时满足：

- 入口在母宇宙
- 返回落回母宇宙
- 世界中能解释这个游戏为什么存在
- 游戏完成后有内容反哺母宇宙

### 4.3 世界内容范围

- 员工角色性格
- 聊天梗沉淀为道具 / 事件 / 对白
- 世界物件注册表
- 游戏入口和退出逻辑
- 可追踪任务和验收汇报

## 5. 非目标

以下内容短期内不允许成为主线目标：

- 再新开无关 demo
- 做一个只能看不能玩的空场景
- 只有剧情没有玩法闭环的小游戏
- 只有玩法没有世界动机的小游戏
- 只在聊天里说“做了”，但没有真实入口
- 只提交局部 bug 修复，却没有推进任何可见内容

## 6. 首发内容要求

## 6.1 第一个完整小游戏：Mushroom Runner

这是第一优先级项目，必须先完整落地。

### 玩法定位

- 2D 平台跳跃
- 轻喜剧像素风
- 角色、道具、障碍都来自 Hermes 宇宙

### 世界动机

- 员工把公司里聊过的“冒充马里奥”梗真正做进宇宙
- 玩家从公司世界进入一段“公司员工 cosplay 平台跳跃”的项目试作
- 游戏不是脱离 Hermes 的独立网页，而是公司项目的一部分

### 必须具备的功能

- 从主世界进入
- 统一命名为 `Mushroom Runner`
- 至少 1 个完整关卡
- 左右移动、跳跃、失败、重开
- 1 个明确终点
- 1 个结算界面
- 安全返回按钮
- 手机端可操作
- 基础音效或视觉反馈

### 可选增强

- 奶茶券 / 锦鲤鳞片 / bug 工单替代金币
- 员工台词作为场景提示
- 关卡中嵌入 Hermes 物件，例如鱼缸、电线、微波炉、办公桌、咖啡机

### 验收标准

- 老板 30 秒内能理解玩法
- 手机和 PC 都能完整玩完一轮
- 不需要刷新页面退出
- 至少有 1 处明确体现 Hermes 世界观的内容

## 6.2 第二个小游戏：柯基追鱼

### 玩法定位

- 轻度追逐 / 闪避 / 收集

### 世界动机

- 柯基和鱼缸是公司办公室真实存在的核心内容，适合做成轻量玩法

### 最低要求

- 从办公室或鱼缸附近进入
- 柯基有明确目标
- 玩家控制柯基避开障碍
- 一轮玩法 30-90 秒
- 可失败，可重开，可返回

## 6.3 第三个小游戏：奶茶外卖

### 玩法定位

- 路线识别 / 角色识别 / 时限配送

### 世界动机

- 奶茶、员工、开会、摸鱼、跑腿是 Hermes 公司日常梗，适合转成轻度模拟玩法

### 最低要求

- 从公司门口或外卖点进入
- 有正确配送目标
- 有时限或错送惩罚
- 员工状态影响送达难度

## 6.4 基础关卡硬要求

现在 4 个员工连基础关卡都没有真正设计出来，这是当前最不能接受的问题之一。

从现在开始，任何小游戏立项后，必须优先交出“基础关卡设计”，否则不算进入开发阶段。

基础关卡设计至少包含：

- 入口位置
- 玩家目标
- 失败条件
- 结束条件
- 关键障碍
- 核心交互
- 关卡节奏
- 手机端操作方式
- 预计时长

没有基础关卡设计，就不允许进入“阿默开始接入和开发”的阶段。

## 7. 世界扩展要求

小游戏不能孤立存在，必须反向影响母世界。

每个小游戏完成后，必须给主世界留下至少 2 项沉淀：

- 一个入口物件或入口区域
- 一条员工讨论或验收反馈
- 一条公司看板任务记录
- 一个可见道具或纪念物
- 一段世界事件摘要

示例：

- `Mushroom Runner` 完成后，公司大厅多一个街机屏或展示海报
- `柯基追鱼` 完成后，鱼缸旁多一个柯基奖牌或高分榜
- `奶茶外卖` 完成后，公司页多一个“外卖配送记录”

### 7.1 言出法随要求

Hermes 像素宇宙必须逐步接近“言出法随”的状态。

这里的含义不是老板一句话立即凭空生成完整游戏，而是：

- 老板提出明确需求后，系统和员工能自动把它转成任务
- 团队在后续 30 到 60 分钟内持续给出可见结果
- 最终结果能够回流为世界变化

世界变化包括：

- 新入口
- 新玩法
- 新道具
- 新对白
- 新任务卡
- 新场景物件
- 新纪念物
- 新宇宙事件

如果聊天结束后什么都没有落到世界里，就不算真正实现了“言出法随”。

## 8. 团队职责

## 8.1 小韩

必须承担“玩法闭环负责人”角色。

交付物不能只是想法，必须包括：

- 玩法目标
- 核心循环
- 关卡结构
- 失败条件
- 完成条件
- 验收标准

禁止行为：

- 只在群里讨论，不输出需求结构
- 把所有方向都说一遍，但不收敛版本

## 8.2 阿默

必须承担“真实可玩交付负责人”角色。

交付物必须包括：

- 可打开入口
- 可玩的版本
- 关键文件路径
- 已知问题
- 下一步

禁止行为：

- 只说“收到”“我去做”
- 只修底层但没有给可验证成果
- 把小游戏做成只能本地临时运行的孤岛页面

## 8.3 小研

必须承担“可玩性与风险负责人”角色。

交付物必须包括：

- 同类参考
- 关键风险
- 参数建议
- 测试清单

禁止行为：

- 只指出问题，不给可验证方案
- 把测试变成空泛吐槽

## 8.4 小文

必须承担“世界观和内容沉淀负责人”角色。

交付物必须包括：

- 角色对白
- 世界入口文案
- 结算文案
- 道具说明
- 聊天梗到游戏内容的映射

禁止行为：

- 只写气氛，不写可落地文本
- 输出与世界无关的空泛设定

## 9. 协作机制

为了避免“员工很被动、每天都在补 bug”，从现在开始必须执行以下规则：

### 9.1 每个版本只能有一个主目标

例如：

- 本周只允许主推 `Mushroom Runner v0.1`
- 其他内容只能作为支援，不得抢主线

### 9.2 每个员工每天至少产出一种可检查成果

- 小韩：需求 / 关卡 / 验收
- 阿默：可玩入口 / 页面 / 接入
- 小研：参数 / 风险 / 测试
- 小文：对白 / UI 文案 / 世界说明

### 9.3 每天必须有“可见推进”

不能只做后台清理。每天至少推进以下之一：

- 新入口
- 新场景
- 新玩法
- 新 UI
- 新文案
- 新任务沉淀

### 9.3.1 每 30-60 分钟交付一次

主线项目推进时，必须遵守以下硬约束：

- 每 30 到 60 分钟至少交付一次可见成果
- 交付要能被老板直接看懂
- 不能连续 1 小时没有任何真实产出

可接受的阶段性交付包括：

- 新版需求文档
- 新入口页面
- 新关卡草稿
- 新玩法参数
- 新测试结果
- 新剧情和 UI 文案
- 新世界沉淀记录
- 新任务拆分与责任归属

不算交付的内容包括：

- “收到”
- “我在看”
- “我先研究一下”
- 纯聊天接梗
- 没有文件、页面、入口、记录支撑的口头承诺

### 9.4 bug 修复不能吞掉主线

Bug 分两类：

- P0 阻塞型 bug：影响访问、聊天、保存、入口、返回，必须优先修
- 非阻塞 bug：登记后排队，不能无限挤占内容开发

要求：

- 每天最多 50% 精力用于 bug
- 至少 50% 精力必须用于新内容推进
- 如果 bug 修复没有直接服务于当前主线交付，就不能无限挤占开发时段

## 10. 任务系统要求

看板不能只显示“待开始 / 处理中 / 阻塞”，还必须体现：

- 这是哪个游戏的任务
- 属于哪个交付阶段
- 当前负责人是谁
- 卡在哪一步
- 下一次汇报节点是什么

推荐统一字段：

- 项目：`Mushroom Runner`
- 模块：`入口 / 玩法 / UI / 文案 / 测试 / 接入`
- 负责人
- 状态：`待开始 / 进行中 / 阻塞 / 待验收 / 完成`
- 验收人：老板
- 输出链接或文件路径

## 11. 验收规则

从现在开始，“完成”必须满足以下条件之一：

### 11.1 文档类完成

- 有真实文件
- 有清晰结构
- 能直接驱动下游开发

### 11.2 功能类完成

- 有入口
- 可操作
- 有结果
- 可返回

### 11.3 世界类完成

- 在主世界可见
- 能被员工识别和提及
- 能进入长期沉淀或任务记录

## 12. 第一阶段交付节奏

建议按下面节奏推进：

### 第 1 阶段：锁定主线

只做一件事：

- `Mushroom Runner v0.1` 需求、入口、可玩闭环、手机端跑通

### 第 2 阶段：补完世界联系

- 游戏入口接入办公室 / Companyverse
- 游戏完成后回流公司世界
- 看板 / 聊天 / 世界事件同步

### 第 3 阶段：扩展第二、第三个小游戏

- 柯基追鱼
- 奶茶外卖

## 13. 老板视角的验收问题

以后每次员工汇报，必须至少能回答下面 6 个问题：

1. 这次推进的是哪个游戏？
2. 它在 Hermes 宇宙里为什么存在？
3. 我现在能从哪里进入？
4. 这一轮我能玩到什么？
5. 还差什么才算完整？
6. 下一步是谁负责？

如果答不出来，就说明还停留在“补 bug”或“空转聊天”，不算真正推进。

## 14. 立即执行要求

从这份需求书生效开始，团队必须执行：

1. 停止继续无目标扩散 demo。
2. 统一首个主线为 `Mushroom Runner v0.1`。
3. 小韩先出玩法闭环和验收标准。
4. 阿默只围绕这个主线做真实可玩入口和接入。
5. 小研出参数、风险、测试清单。
6. 小文补入口文案、关卡提示、结算文案和世界动机。
7. 群聊汇报不允许只有“收到”“我去做”，必须带成果或阻塞说明。
8. 每 30 到 60 分钟至少向老板汇报一次真实交付。
9. 所有交付都要说明它如何沉淀进 Hermes 像素宇宙。

## 15. 一句话结论

Hermes 接下来不是继续“当一个会聊天的像素公司”，而是要变成“一个会持续产出小游戏和世界内容的像素宇宙母世界”。

第一个必须真正落地的目标，不是更多想法，而是：

`Mushroom Runner v0.1 成为一个能从 Hermes 主世界进入、能玩、能结束、能返回、能验收的完整小游戏。`
