## 3. 当前阻塞点

### 3.1 `server.py`

当前存在大规模未提交改动，主要涉及：

- 多账号认证
- `auth.users.json` 配置读取
- 用户级 `company_state.*.json`
- 请求上下文中的当前用户切换
- 登录页 / 注册页逻辑
- 认证 cookie 结构变化

风险：

- 这类修改直接影响 `/login`、`/api/state`、聊天、公司状态读取
- 任何继续在这之上叠加聊天修复，都可能破坏当前公司电脑生产链路

### 3.2 `scripts/core.js`

当前存在中等规模未提交改动，主要涉及：

- 新员工 `linxiaoyan / 小岩`
- 编辑模式网格吸附
- 场景物品坐标吸附与导出
- 新工位 `desk_linxiaoyan`
- 编辑模式提示文案与碰撞逻辑

风险：

- 这部分直接影响办公室布局稳定性、编辑模式实时拖拽与坐标保存
- 如果在未梳理的情况下继续改 `P0 编辑模式`，很容易把当前布局逻辑彻底打散

### 3.3 `scripts/office-scene.js`

当前存在小规模未提交改动，主要涉及：

- 群聊头部员工名单已加入 `小岩`

风险：

- 这个文件本身改动不大，但它与 `core.js`、`chat-ui.js` 共同组成场景聊天行为
- 不应单独假设它是稳定状态

### 3.4 `docs/recovery-checklist-2026-06-26.md`

当前文档也有本地改动。

风险：

- 如果不先固定基线，后续会分不清哪些恢复项是原计划，哪些是本地临时修订

## 4. 当前不适合直接推进的事项

在没有先收口基线前，以下事项都不适合直接发布：

- `P0 主世界办公室恢复稳定`
- `P0 编辑模式最终版恢复`
- `P0 手机单指拖动 / 双指缩放 / 摇杆恢复`
- `P0 聊天实时性`

原因：

- 它们全部依赖当前已经脏掉的 `server.py / scripts/core.js / scripts/office-scene.js`

## 5. 建议的后续收口顺序

### 第一步：认证链路收口

目标：

- 明确当前生产到底使用单账号还是多账号
- 明确 `auth.users.json` 是否正式纳入版本控制
- 明确 `company_state.json` 与 `company_state.*.json` 的职责边界

产出：

- 一次只针对 `server.py` 的收口提交

### 第二步：场景与编辑模式收口

目标：

- 明确网格吸附是否正式启用
- 明确 `linxiaoyan` 是否已经成为正式员工位
- 明确场景物品坐标、导出、碰撞逻辑的最终版本

产出：

- 一次只针对 `scripts/core.js / scripts/office-scene.js` 的收口提交

### 第三步：回到恢复清单继续 P0

优先顺序保持不变：

1. 主世界办公室稳定
2. 编辑模式最终版
3. 手机操作
4. 聊天实时性

## 6. 本次基线审计结论

当前最正确的工程动作不是继续叠加功能，而是先固定并拆清已经存在的运行时改动。

一句话结论：

`现在的真实阻塞不是“功能不会做”，而是“核心运行时文件已经带着未收口改动继续前进，发布风险过高”。`
