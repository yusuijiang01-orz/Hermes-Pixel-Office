## 1. 文档目标

这份蓝图不是愿景海报，而是 Hermes Pixel Universe 接下来进入有序开发的统一施工图。

它要解决 4 个问题：

1. Hermes 当前到底已经有什么。
2. Hermes 还缺什么关键模块。
3. 接下来 2-3 天什么必须先停，什么必须先做。
4. 从 `聊天 -> 游戏素材 -> 宇宙玩法 -> 小游戏矩阵` 的完整链路应该如何落地。

当前判断很明确：

- 项目愿景没有问题，方向也是成立的。
- 当前真正拖慢项目的，不是缺想法，而是缺统一标准、缺模块边界、缺执行顺序。
- 团队必须先从“功能拼接”切到“架构驱动 + 规则驱动 + 版本驱动”。

## 2. 当前统一结论

Hermes 的产品本质，不是单独的聊天工具，也不是单独的小游戏合集。

Hermes 是一个混合体：

- 上层是 `像素公司实时聊天与任务协作系统`
- 中层是 `会沉淀聊天内容的母宇宙世界`
- 下层是 `从宇宙中长出来的小游戏矩阵`

因此，任何新内容都必须回答 3 个问题：

1. 它从哪段聊天或哪条任务长出来？
2. 它在母宇宙里通过什么入口被看见或被进入？
3. 它完成后如何回流为宇宙中的新变化？

如果三者缺一，就不算 Hermes 宇宙内容，只算孤立功能。

## 5. 当前缺失模块盘点

## 5.1 P0 缺失模块：必须先补

### 1. 聊天可靠性治理模块

当前缺：

- 消息唯一性规则不完整
- 群聊线程边界不稳定
- optimistic message 与服务端消息对齐规则不完整
- 用户浏览历史消息时的滚动保护不完整
- 发送失败后的状态反馈不完整

结果：

- 消息重复
- 串台
- 无回复假死
- 用户不信任系统

这是当前头号基建问题。

### 2. 统一世界数据源

当前缺：

- 物件注册表未真正成为唯一来源
- 普通模式、编辑模式、小游戏入口没有统一场景源
- 聊天沉淀内容无法稳定写回世界对象

结果：

- “说加了”不等于“世界里真有”
- 同一物体在不同模式下不一致
- 员工无法稳定感知世界

### 3. 小游戏入口注册表

当前缺：

- 没有标准化入口定义
- 没有“入口 -> 游戏 -> 返回 -> 回流”的固定协议

结果：

- 小游戏容易变成宇宙外孤岛
- 做了也挂不回母世界

### 4. 执行标准与验收标准

当前缺：

- Definition of Done 不清晰
- 基础关卡文档不是强制前置
- 员工每 30-60 分钟交付什么，没有统一标准

结果：

- 大量动作看似忙碌，实际上无法验收

## 6. 核心优先迭代模块

## 6.1 接下来 2-3 天唯一主线

全员暂停无效新功能开发，先打透聊天系统 P0 Bug。

冻结规则：

- 不新增非必要视觉花活
- 不新开无关小游戏
- 不扩写新宇宙区域
- 不做“看起来很酷但不能稳交付”的旁支功能

唯一优先级：

1. 消息丢包
2. 消息重复
3. 线程串台
4. 无状态反馈
5. 浏览历史消息时被强制拉到底部
6. 移动端与桌面端不一致

这一步的目的不是保守，而是先建立用户信任和团队地基。

## 6.2 聊天系统 P0 完成后，立即切换的三个模块

### 模块 A：统一世界数据源

让办公室、编辑模式、Companyverse、小游戏入口都读同一套核心数据。

### 模块 B：小游戏入口注册表

让每个小游戏都能安全地从宇宙进入，再安全返回。

### 模块 C：聊天转宇宙编译器

让聊天不止变任务，还能稳定变成：

- 入口
- 场景物件
- 角色对话
- 支线事件
- 小游戏候选

## 7. 聊天 -> 游戏素材 -> 宇宙玩法 -> 小游戏矩阵 的全链路逻辑

## 7.1 第一段：聊天内容治理

聊天输入分四类：

1. `指令类`
   - 老板明确要求
   - 员工明确承诺
2. `研究类`
   - 查资料
   - 风险提示
   - 参数建议
3. `世界类`
   - 梗
   - 道具
   - 场景想法
   - 角色关系
4. `噪音类`
   - 无落地价值闲聊
   - 重复解释
   - 系统杂讯

输出要求：

- 指令类 -> 必须变任务
- 研究类 -> 必须变研究记录或参数卡
- 世界类 -> 必须变宇宙沉淀候选
- 噪音类 -> 不进长期结构

## 7.2 第二段：素材编译

聊天内容进入编译层后，必须被转成以下五种素材之一：

1. `场景素材`
   - 办公室物件
   - 街区设施
   - 入口装置
2. `叙事素材`
   - 角色对白
   - 结算文本
   - 提示语
3. `玩法素材`
   - 机关
   - 障碍
   - 收集物
   - 胜负条件
4. `系统素材`
   - 任务
   - 参数
   - 规则
5. `宇宙素材`
   - 纪念物
   - 世界事件
   - 后续支线钩子

## 7.3 第三段：宇宙玩法生成

宇宙玩法不是“想到一个小游戏就开页面”，而是按下面方式长出来：

1. 先有世界动机
2. 再有玩法原型
3. 再有入口物件
4. 再有单局闭环
5. 再有结算与回流

标准顺序：

```text
聊天梗 / 老板需求
    ->
世界动机
    ->
玩法原型
    ->
基础关卡
    ->
宇宙入口
    ->
可玩版本
    ->
结算回流
```

## 7.4 第四段：小游戏矩阵

小游戏矩阵不是随便加游戏，而是三层结构：

### 第一批：办公室原生小游戏

- `Mushroom Runner`
- `柯基追鱼`
- `奶茶外卖`

特点：

- 强复用现有资产
- 世界动机清楚
- 进入路径清楚
- 适合快速验证闭环

### 第二批：街区 / Companyverse 小游戏

- 街机台
- 楼道冒险
- 邻里支线

### 第三批：Relicbound 关联玩法

- 从公司项目研发过程反哺到 Relicbound 宇宙
- 形成“游戏中的游戏公司在开发游戏”的套层感

## 10. 今晚之后的执行规则

从今晚开始，团队执行必须遵守以下标准：

1. 没有基础关卡设计，不准开小游戏开发。
2. 没有入口定义，不准说小游戏已接入宇宙。
3. 没有返回路径，不准说小游戏可玩。
4. 没有回流结果，不准说内容已完成。
5. 没有文件路径、任务记录、验收口径，不准说交付成立。

## 12. 蓝图结论

Hermes 下一阶段最重要的不是“再想一个更大的世界”，而是先把这条生产线跑顺：

`聊天可信 -> 任务可追 -> 世界可变 -> 入口可进 -> 游戏可玩 -> 结果可回流`

只要这条线跑通，像素宇宙就会真的开始生长。

只要这条线跑不通，世界再大也只是空壳。
