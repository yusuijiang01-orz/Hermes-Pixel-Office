# Hermes Auth Boundary Audit

生成时间：2026-06-30 15:14

目的：

- 收口 `server.py` 中已经出现的多账号 / 注册 / 独立公司状态逻辑
- 明确哪些改动可以保留，哪些必须隔离，哪些暂时不应继续叠加
- 为下一步恢复 `P0 聊天实时性 / 公司状态 / 登录稳定性` 提供边界

## 1. 当前观察到的认证改动

当前 `server.py` 已经不再是单纯的单账号 Basic Web Login，它已经长出了一套更复杂的账号体系，主要包括：

- `AUTH_USERS_PATH = ROOT / "auth.users.json"`
- `configured_auth_users()`
- `default_auth_username()`
- `REQUEST_CONTEXT` / `current_request_username()` / `set_request_username()`
- `run_as_user()`
- `current_user_config()`
- `current_company_state_path()`
- `load_company_state(path=None, username=None)`
- `save_company_state(data, path=None, username=None)`
- `make_auth_cookie(username)`
- `verify_auth_cookie(cookie_header)` 返回用户名，而不是 `bool`
- 自定义 `login_page(...)`
- 自定义 `register_page(...)`
- `validate_registration(...)`
- `register_auth_user(...)`

这说明当前代码已经从：

- “一个固定网页登录口”

演进到了：

- “一个支持注册、多账号、每个账号独立公司状态和独立任务板的系统”

## 2. 当前认证模型的真实含义

按现有代码，用户配置来自两层：

### 2.1 默认账号层

来自环境变量：

- `WEB_AUTH_USER`
- `WEB_AUTH_PASSWORD`

这层相当于“默认老板账号 / 默认公司账号”。

### 2.2 扩展账号层

来自：

- `auth.users.json`

这层允许新增用户，并为每个用户指定：

- `password`
- `board`
- `company_state`
- `studio_name`

也就是说，这不是单纯“多个人共用一个公司”，而是“每个人可以有自己的独立 Hermes 公司实例”。

## 3. 当前模型的直接风险

### 3.1 认证链路和运行态链路耦合过深

当前多账号逻辑不仅影响登录，还影响：

- 公司状态文件路径选择
- 当前任务板选择
- API 状态返回
- 最近事件 / 待办 / 沉淀的读写位置

这意味着：

- 认证问题不是“登录页坏了”
- 认证问题会直接扩散成“聊天串台 / 看板串状态 / 任务落错公司”

### 3.2 注册逻辑已经是产品级功能，不是临时脚本

`register_auth_user(...)` 当前会自动做这些事：

- 写入 `auth.users.json`
- 创建独立 board
- 创建独立 `company_state.<slug>.json`
- 注入欢迎通知
- 注入独立公司最近事件

所以它已经不是“方便测试的小入口”，而是正式产品能力。

风险在于：

- 如果这个能力没有明确产品决策，就不应该继续半完成地挂在生产入口上

### 3.3 `verify_auth_cookie()` 语义已经变了

之前：

- 登录校验像布尔值

现在：

- 登录校验结果是用户名字符串

这会影响所有把它当布尔值或把“当前用户”当隐式默认用户的地方。

如果外围调用没完全跟上，就会出现：

- 已登录但状态不一致
- API 读到别人的公司状态
- 登录态存在，但页面逻辑拿错用户

## 4. 建议保留的部分

以下改动有明确价值，建议保留：

### 4.1 `configured_auth_users()`

原因：

- 它把默认账号与 `auth.users.json` 合并为同一读取入口
- 这是多账号最基础、也最合理的配置入口

### 4.2 `current_user_config()` / `current_company_state_path()`

原因：

- 它们把“当前用户对应哪个公司状态文件”显式化了
- 这比把状态路径硬编码死在全局变量里更正确

### 4.3 自定义网页登录页

原因：

- 这个项目已经明确不想继续用浏览器原生认证弹窗
- 像素风登录页是合理产品方向

但前提是：

- 登录页稳定优先于注册页

## 5. 建议隔离的部分

以下改动不是不能存在，而是应该先隔离：

### 5.1 `register_page(...)` 和 `register_auth_user(...)`

原因：

- 注册是二级能力，不是当前恢复主线
- 当前主线是恢复已有老板公司，而不是先做多租户平台

建议：

- 先保留代码
- 但不要继续扩展注册产品逻辑
- 先把登录、当前老板账号、当前公司状态跑稳

### 5.2 独立 board 自动创建逻辑

原因：

- 一旦注册与 board 绑定，就会把问题带入任务系统
- 当前 recovery 重点是本公司的聊天、场景、看板恢复

建议：

- 允许保留现有实现
- 但后续不要把 recovery 修复建立在“多用户都要同时工作”这个假设上

## 6. 建议暂缓继续叠加的部分

### 6.1 再往 `server.py` 里加聊天实时性补丁

不建议原因：

- 认证边界还没收口
- 任何聊天补丁都可能落到错误用户状态文件里

### 6.2 再修改 `/login` / `/register` 的交互页面

不建议原因：

- 现在缺的不是视觉，而是认证边界稳定性

### 6.3 再改 `company_state.*.json` 路由策略

不建议原因：

- 如果没有先确认“生产到底是单公司还是多公司”，继续改只会制造更多分叉状态

## 7. 推荐的收口方案

下一阶段应把认证收口分成两步。

### 第一步：定义当前生产模式

必须先定清楚：

- 当前生产是否只服务老板这一个公司
- 还是已经正式支持“每个登录账号一个独立公司”

如果答案是“当前生产只服务老板这一个公司”，则应：

- 保留登录页
- 保留 `configured_auth_users()`
- 保留 cookie 登录
- 但暂时把注册与多公司能力降级为非主路径

如果答案是“当前生产正式支持多公司”，则应：

- 把 `auth.users.json`
- `company_state.<user>.json`
- board 分配逻辑
- `/api/state`

全部做一次完整一致性验证。

### 第二步：收口到单独提交

认证收口提交只允许触碰：

- `server.py`
- `auth.users.json`（如果决定纳入版本控制）
- 相关说明文档

不应同时混入：

- 聊天 UI 修复
- 场景坐标修复
- 编辑模式修复
- 柯基或小游戏修复

否则后续无法定位问题来源。

## 8. 本次审计结论

当前最重要的事实不是“已经有多账号”，而是：

`多账号认证已经开始影响公司状态与聊天状态文件选择，所以它必须先被定义边界，不能继续作为隐性前提存在。`

一句话结论：

`先决定当前生产是“单老板公司”还是“多用户独立公司”，再继续做聊天与运行态修复。`
