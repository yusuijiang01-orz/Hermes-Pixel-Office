# Hermes Production Auth Mode Decision

生成时间：2026-07-01

状态：

- 生效范围：当前生产环境
- 生产域名：`pix.lovenom.eu.org`
- 生产链路：`Cloudflare Tunnel -> 公司电脑 -> 127.0.0.1:8777`

## 1. 结论

当前生产模式应明确为：

`单老板公司模式`

这意味着：

- 当前生产只服务老板这一个 Hermes 公司实例
- 当前生产不把“多用户独立公司”作为正式能力对外承诺
- 现有多账号 / 注册 / 独立公司状态逻辑可以继续存在于代码中，但不得继续作为当前恢复工作的默认前提

## 2. 为什么必须这样定

### 2.1 当前恢复目标不是做多租户平台

现在的主线是：

- 恢复聊天稳定性
- 恢复公司状态一致性
- 恢复办公室 / 编辑模式 / 手机操作
- 恢复任务与宇宙沉淀闭环

这些都属于“把当前这一个老板公司的生产环境修稳”。

不是：

- 同时支持多个注册用户
- 同时支持多个独立公司
- 同时保证多 board、多状态文件、多账号会话隔离

### 2.2 多账号逻辑已经开始影响运行态

当前 `server.py` 中，多账号并不只是登录页功能，而是已经影响：

- `company_state` 路径选择
- board 选择
- `/api/state`
- cookie 用户识别
- 最近事件、任务、沉淀的读写落点

如果不先把生产模式定为单老板公司，那么后续任何聊天或状态修复都可能继续出现：

- 串台
- 读错状态
- 看板内容来自错误公司
- 登录正常但数据不一致

### 2.3 当前域名和部署方式也说明了它应先是单公司

当前公开入口只有一个：

- `pix.lovenom.eu.org`

当前部署目标也只有一套：

- 这台公司电脑
- 这一个本地 Hermes 服务

这更符合：

- “一个正在恢复中的主公司环境”

而不符合：

- “一个已经准备好面向多租户用户开放注册的平台”

## 3. 这项决策的直接约束

从现在开始，直到聊天与状态恢复完成前，默认遵循以下规则。

### 3.1 生产恢复默认只考虑老板主账号

后续恢复、验证、部署、验收默认只围绕：

- 当前老板登录入口
- 当前主公司状态
- 当前公开域名

展开。

### 3.2 注册能力不是当前恢复主路径

可以保留：

- 注册页面代码
- `auth.users.json` 支持
- 独立 `company_state.<user>.json` 机制

但当前不应：

- 继续扩展注册流程
- 继续新增多公司产品逻辑
- 把 recovery 修复建立在“多账号都要同时正确工作”的前提上

### 3.3 聊天与状态修复必须先以单公司一致性为准

后续任何运行态修复应优先保证：

- 同一账号进入后，状态稳定
- 同一域名下，PC 和手机看到的是同一家公司
- 群聊、私聊、看板、公司状态来自同一套数据

只有这一步稳定了，才有资格继续评估多公司能力。

## 4. 对代码收口的影响

这项决策并不要求立刻删掉多账号代码，但要求后续实现遵守边界。

### 4.1 可以保留

- `configured_auth_users()`
- 自定义登录页
- cookie 登录
- `current_user_config()`
- `current_company_state_path()`

### 4.2 应降级为非主路径

- 注册页
- 自动创建独立 board
- 自动创建独立公司状态文件

### 4.3 后续 runtime 修复默认以主账号为基准验证

例如后续修复：

- `/api/state`
- 聊天发送
- 看板聚合
- 最近交付
- 世界沉淀

都应先验证：

- 主账号是否稳定

而不是先验证：

- 多账号是否同时可用

## 5. 后续执行顺序

这项决定落地后，建议顺序如下：

1. 以单老板公司模式为前提，收口 `server.py` 认证路径
2. 保证当前主账号登录后，公司状态唯一且稳定
3. 再继续做 `P0 聊天实时性`
4. 再继续做 `P0 主世界 / 编辑模式 / 手机操作`
5. 聊天和状态链路稳定后，再重新评估多账号 / 多公司是否保留为正式能力

## 6. 一句话结论

`当前生产不是多租户平台，而是一个需要先被修稳的老板主公司环境。后续所有恢复动作，都应先服务这一个公司。`
