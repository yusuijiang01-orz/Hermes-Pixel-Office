# Hermes Pixel Office 上下文与优化路线

生成时间：2026-06-29

## 当前目标

Hermes Pixel Office 不是纯聊天皮肤，而是一个可以被观察、编辑、协作和扩展的像素游戏公司。老板通过网页或未来 iPhone 客户端进入公司，员工既能聊天，也能感知环境、执行任务、推进游戏项目。

## iPhone 客户端方案

可以单独制作 iPhone 客户端，建议先做 iPhone，安卓后置。

第一版不要重写一套游戏逻辑。推荐做 SwiftUI + WKWebView 的原生壳：

- 首屏：Base URL、API Key、账号登录。
- 登录状态：使用 Keychain 保存 token，不让用户每次输入账号密码。
- 渲染：继续加载同一套 Hermes 网页端，保证与网页模式一模一样。
- 原生能力：全屏、禁用 Safari 地址栏干扰、触觉反馈、推送通知、后台刷新。
- 后续再把 Canvas 渲染层抽成 SpriteKit 或 Metal，不在第一版里硬拆。

这样能最快实现“像 App 一样打开”，同时不产生网页和客户端两套世界不同步的问题。

## 当前已确认问题

- 编辑模式后部分动画被静态绘制覆盖：鱼缸、水草、水吧等。
- 书柜曾经用随机数逐帧绘制，导致看起来会动。
- 手机点员工直接进聊天，老板看不到员工状态。
- 聊天返回按钮应该回到公司 UI 主界面，而不是在聊天页循环。
- 柯基状态切换太频繁，像动画播放器，不像真实狗。
- 群聊未完成消息会显示一堆“正在输入”，造成假活跃感。
- 微波炉等物品可能只存在于聊天记忆，不一定进入世界物品注册表。
- 员工行为过多依赖固定路线，而不是查询环境物品坐标。
- 公司与现实公司差距明显：缺少稳定作息、位置感、物品交互、任务执行反馈、员工之间自然接话。

## 已开始修复的点

- 群聊恢复显示。
- 私聊恢复显示。
- 微波炉进入世界物品注册表。
- 前端开始上报办公室物品坐标到 `/api/world/report`。
- 员工上下文可读取 `world_objects`。
- 鱼缸编辑模式下重新使用动画绘制函数。
- 群聊中未完成消息不再直接显示“正在输入”假气泡。
- 手机点击员工先选中员工并显示状态，不再直接跳聊天。
- 手机聊天返回按钮回到公司页。
- 柯基状态切换改为低频状态机：大部分时间 idle/sleep，少量 bark。

## 环境感知设计

每个世界物品必须注册为结构化对象：

```json
{
  "id": "microwave",
  "name": "微波炉",
  "scene": "office",
  "x": 1584,
  "y": 479,
  "w": 32,
  "h": 36,
  "category": "facility",
  "owner": "company",
  "interactable": true,
  "dynamic": false,
  "affordances": ["heat_food", "toast"]
}
```

员工行动前应查询：

- 我在哪个场景。
- 我附近有什么。
- 目标物品在哪里。
- 目标物品是否可交互。
- 是否有路径可以到达。
- 如果物品移动了，是否需要更新工作/生活目标。

例子：

- 小文想摸柯基：查询 `corgi` 坐标，规划路径，移动到附近，触发 pet 行为。
- 员工渴了：查询 `breakbar` 或水瓶坐标，走过去喝水。
- 老板问微波炉在哪：员工读取 `world_objects.office.microwave`，回答房间和坐标。
- 办公桌被编辑移动：员工查询自己的 `desk_*` 坐标，重新回到新工位。

## 群聊交互升级

目标不是 4 个人同时贴几段，而是像真实群聊：

- 老板发话题后，小韩先做主管拆解。
- 阿默根据技术可行性回应。
- 小研补充证据或反驳。
- 小文补体验和文案。
- 每条消息按真实完成时间一条一条出现。
- 没发出来之前不在聊天流显示“正在输入”刷屏。
- 可以有一个很轻的顶部状态：小韩正在组织回复，但不要把它当消息气泡。
- 员工之间应引用前一句或 @ 某人，而不是各说各的。

## 与现实公司的差距

- 缺少空间记忆：员工不真正知道桌子、厕所、水吧、鱼缸在哪。
- 缺少物品状态：微波炉、咖啡机、鱼缸、柯基没有统一状态源。
- 缺少路径规划：柯基和员工容易穿墙或固定路线。
- 缺少生理节律：疲劳、饥饿、口渴、心情对行为影响还不够强。
- 缺少任务闭环：聊天承诺和实际文件改动还没有稳定绑定。
- 缺少团队协作节奏：群聊还不够“你一句我一句”。
- 缺少场景栈：小游戏、厕所、老板办公室等容易进去出不来。
- 缺少客户端一致性：网页、VPS、未来 iPhone 客户端需要同一套版本与缓存策略。

## 下一步优先级

1. 建立世界物品注册表为单一事实源，普通模式、编辑模式、员工感知共用。
2. 给员工增加环境查询工具：按场景、按物品 id、按 owner 查询物品与坐标。
3. 给员工/柯基增加简单路径规划，先避免穿墙和穿桌。
4. 群聊调度改为回合制队列：一条消息完成后再触发下一位。
5. 手机 UI：员工状态卡、聊天入口、返回公司页彻底分离。
6. iPhone 客户端第一版：SwiftUI 设置页 + WKWebView + Keychain 保存配置。
7. 全局场景栈：任何场景都有返回和 Home。
